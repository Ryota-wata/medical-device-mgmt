# 第5章：プロジェクト概要のインプット

## 5.1 インプットの目的と位置づけ

### 本章の位置づけ

```
第5章（本章）         第6章              第7章
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ PJ概要      │ →  │ 要件定義    │ →  │ 画面設計    │
│ インプット  │    │             │    │             │
└─────────────┘    └─────────────┘    └─────────────┘
  ↓                  ↓                  ↓
・業務概要          ・ユーザー一覧      ・画面一覧
・現状と課題        ・ユースケース      ・画面モック
・対応デバイス      ・機能一覧          ・画面設計書
・デザイン方針      ・非機能要件
```

### インプットすべき情報

本章では「顧客から得た生の情報」をClaude Codeに伝えます。
ユーザー一覧や機能一覧は次章（要件定義）でClaude Codeに生成させます。

| インプット項目 | 内容 | 必須 |
|--------------|------|:----:|
| システム名・目的 | 何をするシステムか | ◎ |
| 対象業務 | どの業務範囲をカバーするか | ◎ |
| 現状と課題 | 現在の運用方法と問題点 | ◎ |
| 対応デバイス | PC/タブレット/スマホ | ◎ |
| デザイン方針 | テイスト、参考サイト | ○ |
| 技術スタック | フレームワーク指定がある場合 | △ |

## 5.2 基本情報のインプット

### プロンプトテンプレート

```
これからモック駆動で画面設計を進める。
まず以下のプロジェクト概要を理解して。

【システム名】
[システム名]

【システムの目的】
[何を実現するシステムか、1-2文で]

【対象業務】
[このシステムがカバーする業務の範囲]

【対応デバイス】
- PC: [対応/非対応]
- タブレット: [対応/非対応]
- スマートフォン: [対応/非対応]
```

### 実行例

```
これからモック駆動で画面設計を進める。
まず以下のプロジェクト概要を理解して。

【システム名】
医療機器管理システム

【システムの目的】
病院内の医療機器を一元管理し、資産管理業務を効率化する

【対象業務】
- 医療機器の資産管理（登録・変更・廃棄）
- 棚卸
- 修理依頼・管理
- 点検管理
- QRコードによる機器識別

【対応デバイス】
- PC: 対応（管理者向けメイン操作）
- タブレット: 対応（棚卸作業）
- スマートフォン: 対応（QRスキャン・棚卸）
```

## 5.3 現状と課題のインプット

### プロンプトテンプレート

```
現状の運用方法と課題を伝える。
これを解決するシステムを設計してほしい。

【現状の運用方法】
[どのように業務を行っているか]

【課題】
1. [課題1]
2. [課題2]
3. [課題3]

【期待する改善効果】
[システム導入で何が改善されるか]
```

### 実行例

```
現状の運用方法と課題を伝える。
これを解決するシステムを設計してほしい。

【現状の運用方法】
- 医療機器の管理はExcelで行っている
- 棚卸は紙のチェックリストを使用
- 修理依頼はメールと電話で行っている

【課題】
1. Excelファイルが複数あり、最新がどれかわからない
2. 棚卸に1週間以上かかり、その間業務に支障が出る
3. 修理履歴が追えず、同じ故障を繰り返す機器が特定できない
4. 機器の設置場所が把握できていない

【期待する改善効果】
- データの一元管理で情報の正確性向上
- QRコードによる棚卸の効率化（3日以内に短縮）
- 修理履歴の可視化による予防保全
```

## 5.5 既存資料がある場合のインプット

### 資料を読み込ませるプロンプト

```
以下のファイルにプロジェクトの情報がある。

[ファイルパス]

```

### 資料から情報を抽出させるプロンプト

```
読み込んだ資料から以下を抽出して。

- システムの目的
- 対象業務の範囲
- 現状の課題
- 期待する効果

箇条書きで整理して。
```

## 5.6 インプット完了の確認

### 理解度確認プロンプト

インプットが完了したら、Claude Codeの理解度を確認します。

```
ここまでのインプット内容を要約して。
理解した内容に誤りや不足があれば指摘して。
```

### 次ステップへの移行プロンプト

理解度を確認後、次章（要件定義）へ進みます。

```
インプット内容の理解はOK。
次は要件定義を進める。
まず、このシステムを使うユーザーの種別を洗い出して。
```

> **ポイント**: このプロンプトで第6章の作業が始まります。
> 生成されたユーザー一覧を基に、ユースケース→機能一覧→画面一覧と段階的に生成していきます。

## 5.7 よくある質問

### Q: どこまで詳細にインプットすべき？

**A**: 以下の情報があれば十分です。
- システムの目的（1-2文）
- 対象業務の範囲（箇条書き）
- 現状の課題（3-5項目）
- 対応デバイス

機能一覧やユーザー一覧は、次章でClaude Codeに生成させます。

### Q: 顧客資料がそのまま使えない場合は？

**A**: 資料を読み込ませた後、以下のプロンプトで整理させます。

```
読み込んだ資料は情報が散在している。
モック開発に必要な情報を整理して、
以下の形式でまとめ直して。

- システム概要
- 対象業務
- 現状の課題
```

### Q: 技術スタックの指定は必要？

**A**: 特に指定がなければ、CLAUDE.mdに記載した技術スタック（Next.js + TypeScript等）が使用されます。
顧客から特定の技術指定がある場合のみ伝えてください。
