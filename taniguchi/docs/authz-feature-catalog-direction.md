# 認証認可 再設計方針メモ（機能カタログ・施設単位制御）

本メモは、2026-04-01 のクライアント合意内容を受けて、認証認可の見直し方針を整理するための作業メモである。  
現時点では [機能要件.md](/C:/Projects/mock/medical-device-mgmt/taniguchi/機能要件.md) と [db-schema.puml](/C:/Projects/mock/medical-device-mgmt/taniguchi/db/db-schema.puml) の正本更新前の整理資料として扱う。

## 背景

- これまでの認証認可設計は、`roles` / `role_permissions` / `role_facility_scopes` / `user_accessible_facilities` を前提としていた。
- ただしクライアントとの再協議により、「事前定義ロールを基準に権限を配る」考え方をやめ、施設ごとの機能提供可否と、施設に紐づくユーザーごとの利用可否を管理する方向へ切り替える。
- あわせて、他施設データの閲覧についても、設立母体に依存しない任意グループと、公開対象データ／公開対象カラムの制御を持つ必要があることが確認された。

## クライアント要望

### 1. ロール前提の権限制御を廃止する

- 事前定義ロールを正本とする考え方は廃止する。
- 施設単位で「どの機能を提供するか」を ON/OFF で管理する。
- 施設に紐づくユーザーは、その施設で提供されている機能のうち、どれを利用可能にするかを ON/OFF で管理する。

### 2. 任意グループによる他施設閲覧を導入する

- 設立母体を問わずに施設をグルーピングできる任意グループを導入する。
- 自施設が含まれる任意グループに属する契約中の他施設について、データ閲覧を可能にする。
- ただし、他施設ユーザーに対して何のデータ閲覧を許可するかは定義できなければならない。
- 一覧画面で表示される特定種類のカラムについても、提供するか否かを ON/OFF で切り替えられるようにする。

### 3. SHRCユーザーは施設単位で担当と利用機能を管理する

- SHRCユーザーには担当施設をアサインする。
- 担当施設ごとに、その施設で利用可能な機能を管理できるようにする。
- SHRC専用の別系統ロールではなく、施設単位の割当で吸収する。

### 4. クライアント整理は機能カタログとして扱う

- クライアントサイドで整理された一覧は、今後の認可設定で扱う「機能カタログ」の元データとして扱う。
- その中には、画面利用、主要操作、メニュー表示、一部カラム表示制御が含まれている。
- 例:
  - 機能カタログ候補: `QRコード発行`、`点検管理`、`原本リストの修正追加`
  - カラム制御候補: `原本価格情報のカラム項目`、`DataLINKのSHIPのみカラム項目`

## 方針

### 1. 権限モデルの主語を「ロール」から「施設提供機能」へ置き換える

- 既存の `roles` / `role_permissions` を認可判定の正本とする前提は採用しない。
- 認可判定の主語は、以下の 3 層または 4 層で整理する。
  - ユーザーがどの施設を担当・作業対象にできるか
  - 施設がどの機能を提供しているか
  - ユーザーが担当施設ごとにどの機能を使えるか
  - 他施設へどのデータ／どのカラムを公開するか

### 2. 「機能カタログ」と「カラムカタログ」を分けて扱う

- 画面利用、主要操作、出力、モーダル起点業務などは「機能カタログ」として管理する。
- 一覧や詳細の一部表示項目は「カラムカタログ」として管理する。
- 認可設定では、機能 ON/OFF とカラム ON/OFF を分けて扱えるようにする。
- カラム制御は、特に他施設閲覧時の公開範囲制御で利用する。

### 3. 施設単位の設定とユーザー単位の設定を分離する

- 施設単位では「その施設の所属・担当ユーザーへ提供する機能」を管理する。
- ユーザー単位では「施設で提供されている機能のうち、そのユーザーへ許可する機能」を管理する。
- ユーザー単位の設定は、施設単位設定を超えないことを前提とする。

### 4. 他施設閲覧は公開ポリシーとして別管理する

- 他施設閲覧は、単に「施設にアクセスできるか」ではなく、公開元施設が何を見せるかを持つ必要がある。
- そのため、任意グループ所属だけでなく、以下を別途持つ前提で整理する。
  - 他施設へ公開可能なデータ種別
  - 他施設へ公開可能なカラム種別
- 現行の `facility_asset_view_groups` / `facility_asset_view_group_members` は、`facility_collaboration_groups` / `facility_collaboration_group_facilities` へ改称したうえで任意グループの土台として再利用余地があるが、公開対象定義を追加しなければ要件を満たさない。

### 5. SHRCユーザーは施設単位割当へ統合する

- SHRCユーザー専用のロールモデルや `scope_type` ベースの施設判定を拡張するのではなく、担当施設割当と施設単位機能設定へ統合する。
- これにより、SHRCユーザーの「施設Aでは利用可、施設Bでは不可」のような差分を同一モデルで表現できる。

### 6. `feature_code` はカタログ識別子として継続利用する

- `feature_code` という識別子自体は継続利用できる。
- ただし、格納先は `role_permissions.feature_code` 前提ではなく、施設機能設定・ユーザー施設機能設定・他施設公開設定で参照するカタログコードとして再定義する。
- 既存の [feature-code-policy.md](/C:/Projects/mock/medical-device-mgmt/taniguchi/docs/feature-code-policy.md) は、命名規則や粒度の考え方は参考になるが、ロール前提の説明は見直し対象とする。

## 現時点の設計整理

### 認可設定で必要になる管理対象

- ユーザー担当施設
  - ユーザーが作業対象として選択できる施設
  - SHRCユーザーの担当施設もここで表現する
- 施設提供機能
  - 施設がその施設の所属・担当ユーザーへ提供する機能
- ユーザー施設機能
  - 特定ユーザーが、特定施設で利用できる機能
- 他施設閲覧グループ
  - 設立母体非依存の任意施設グループ
- 他施設公開データ設定
  - 他施設ユーザーへ公開するデータ種別
- 他施設公開カラム設定
  - 他施設ユーザーへ公開するカラム種別
- 認証前提機能
  - ログイン、パスワード再設定は認証基盤として扱い、施設単位・ユーザー単位の ON/OFF 管理対象には含めない
- 施設選択
  - 作業対象施設の選択は `user_facility_assignments` で割り当てられた担当施設一覧から行い、`facility_select_all` は認可機能として採用しない
- 価格カラム
  - `original_price_column`、`own_price_column`、`other_price_column` はクライアント整理が分かれているため、文脈別に分離維持する
- 表示分類
  - `users.account_type` を保持し、認可判定には使わず、表示分類や入力制御用途に限定する

### 認証認可APIで見直しが必要な点

- `GET /auth/me`
  - `role` / `permissions` / `scope_type` 前提ではなく、ログインユーザー基本情報、担当施設一覧、既定施設などを返す形へ見直す必要がある。
- `GET /auth/context`
  - 施設選択後の画面表示制御用に、`actingFacilityId` を受けて当該施設における実効機能と実効カラムを返す形を前提とする。
- `POST /authorization/check`
  - `role_permissions` を引くのではなく、対象施設における施設提供機能・ユーザー施設機能・他施設公開ポリシーをもとに判定する必要がある。
- 業務API
  - `facilityId` を受けるAPIは、対象施設に対する機能利用可否と、他施設閲覧時の公開範囲の両方を判定する必要がある。
