# API設計書 記載テンプレート

最終更新: 2026-03-13

本書は、[API設計書_SHIP部署マスタ.docx](/C:/Projects/mock/medical-device-mgmt/taniguchi/api/Fix/API設計書_SHIP部署マスタ.docx) を基準に、今後 API 設計書を作成・レビューする際の章立てと記載項目を整理したテンプレートである。  
特に CRUD 系の画面、マスタ管理系の画面、一覧/登録/更新/削除を持つ画面では、このテンプレートを標準形として扱う。

認証/認可のような横断 API は一部の章や記載粒度が異なってよいが、少なくとも「文書概要」「API の位置づけ」「共通仕様」「API 一覧」「機能設計」「権限/業務ルール」「エラー」「運用方針」は同じ考え方で揃える。

## 書式ルール
- フォントは全文 `Meiryo UI` を使用する。
- 見出しはすべてボールドとする。
- 見出し1は `20pt` とする。
- 見出し2は `16pt` とする。
- 見出し3は `14pt` とする。
- 見出し4は `12pt` とする。
- 標準本文は `11pt` とする。
- テーブルの最上位行（カラム名行）はボールドとする。
- テーブルの最上位行（カラム名行）の塗りつぶしは `青, アクセント1, 白+基本色80%` とする。

| 項目名 | 型 | 説明 |

### エラーコード一覧
| エラーコード | HTTPステータス | 内容 | 発生条件 |
| --- | --- | --- | --- |
| `NOT_FOUND` | `404` | 対象データが存在しない | 指定IDが未存在 |
| `DUPLICATE` | `409` | 一意制約に抵触 | 名称重複 |
| `MASTER_IN_USE` | `409` | 参照中のため削除不可 | 関連データが存在 |
| `VALIDATION_ERROR` | `400` | 入力値不正 | 必須不足、桁数超過 |
| `INTERNAL_SERVER_ERROR` | `500` | サーバー内部エラー | 想定外エラー |

## 1. 基本方針
- 画面要件の正本は [機能要件.md](/C:/Projects/mock/medical-device-mgmt/taniguchi/機能要件.md) とする。
- テーブル定義の正本は [db-schema.puml](/C:/Projects/mock/medical-device-mgmt/taniguchi/db/db-schema.puml) とする。
- API 設計書では、画面から見た操作と、保存先/参照先テーブル、権限判定、エラー条件を対応づけて記載する。
- モック実装に依存する一時的な UI 挙動は、要件で Fix していない限り API 設計に持ち込まない。
- 1 本の API 設計書で複数画面を扱う場合は、対象画面と役割分担を第1章または第2章で明示する。

## 第1章 概要

### 1.1 本書の目的
- 対象機能に対して、どの API をどう使うかを明記する。
- 画面要件、DB 設計、API の責務を一致させる目的を簡潔に書く。

記載例:
- 本書は SHIP部署マスタ画面で利用する API の仕様を定義する。
- 画面操作とデータ更新の対応関係を明確にし、実装とレビューの基準を統一する。

### 1.2 対象システム概要
- 対象システム名
- この機能が業務全体のどこで使われるか

記載例:
- 対象システムは医療機器管理システムである。
- 本機能は共通マスタ管理の一部として利用される。

### 1.3 用語定義
- 文書内で繰り返し使う用語だけを簡潔に定義する。
- 画面固有の言い換えがある場合はここで揃える。

記載すべき項目例:
- 機能名
- マスタ名
- 対象データの意味
- 文書内の略称

### 1.4 対象画面
- API 設計書が対応する画面を一覧化する。
- 複数画面を1本で扱う場合は、画面ごとの役割も併記する。

推奨形式:
- 画面名
- 画面パス
- 利用目的

## 第2章 システム全体構成

### 2.1 API の位置づけ
- その API 群がシステムのどこに位置するかを書く。
- 他画面・他 API とどうつながるか、ここで先に説明する。

記載例:
- 共通マスタの参照・登録・更新・削除を担う。
- 個別部署マスタ画面の候補データとしても利用される。

### 2.2 画面と API の関係
- 画面のどの操作でどの API を使うかを対応づける。
- 一覧、検索、作成、更新、削除、モーダル表示後の保存などを抜けなく書く。

推奨形式:
- 画面操作
- 呼び出す API
- 補足

### 2.3 使用テーブル
- 参照テーブル
- 更新テーブル
- 補助テーブル
- 参照専用 VIEW

最低限書くこと:
- テーブル名
- 利用種別（READ / CREATE / UPDATE / DELETE）
- 用途

## 第5章 機能設計

API ごとに以下の粒度を揃える。  
これが実装者・レビュー担当が一番見る章なので、省略しない。

### 5.x API 名

#### 冒頭説明
- API の目的を1〜2文で書く
- 何を取得/登録/更新/削除するかを先に説明する

#### リクエストパラメータ / リクエストボディ
- 一覧系は `リクエストパラメータ`
- 登録更新系は `リクエストボディ`
- 必要に応じて Path Parameter もここで明記する
- 各項目の型、必須、説明を表で書く

#### 権限チェック
- 必要な `feature_code`
- 必要な `permission_level`
- 想定ロール
- 施設スコープや契約状態を使うならその条件

#### 処理仕様
- 何を参照するか
- 何を保存するか
- 一意性チェック
- 参照整合性チェック
- 削除可否判定
- どの順番で処理するか

#### レスポンス
- 正常時の HTTP ステータスとレスポンス型名を書く
- 項目一覧を表で書く
- ネストした要素がある場合は `items要素` や `item要素` として続けて書く

#### ステータスコード
- HTTP
- 条件
- Body
