## 確定事項

### 編集リストの役割
- 編集リストは**誰でも何回でも作成できる作業用リスト**
- 見積依頼No付与 → 購入管理タスク画面で見積アップロード → 原本登録の流れは**確定**

### リモデル管理と編集リストの分離方針
- リモデル管理（進捗管理・処理区分決定）は編集リストとは**別の画面・別のデータ構造**で行う
- 編集リストは見積依頼ワークフロー専用ツールに限定する
- 理由: 編集リストは複数存在しうるため「どれが公式か」が曖昧になる

### 原本資産の参照方式
- リモデル管理画面は原本資産を**ライブ参照**（読み取り専用）
- 処理区分・新場所の情報は原本とは別のデータ構造（remodelActions）に保存
- 原本への書き戻しは**クローズ時のみ**（移設資産の新場所反映）
- 編集リストは従来通りスナップショット（コピー）で問題なし

### リモデルプロジェクトの単位
- 1病院単位ではなく**プロジェクト単位**
- 複数病院の統合移転にも対応する（対象施設: 複数可）

### 処理区分の定義

| 処理区分 | 意味 | 後続フロー |
|---------|------|-----------|
| 未決定 | まだ判定されていない（初期状態） | - |
| 移設 | 新施設にそのまま移動 | 施設マスタから新場所マッピング → クローズ時に原本反映 |
| 更新 | 買い替え | → 編集リストに追加 → RFQ → 見積 → 発注 → 検収 → 資産登録 |
| 廃棄 | 処分 | → 廃棄管理タスクに連携 |
| 保留 | 判定保留（コメント入力可） | 定期的に見直し |

## 未決定事項

### 1. 編集リストの作成単位
- 更新判定→編集リスト作成、増設→編集リスト作成、新規申請→編集リスト作成と個別に作ると負荷が高い
- **案A**: プロジェクトに1つの編集リストを紐づける
- **案B**: 既存リストに追加 or 新規作成を選択するモーダル
- → **要決定**

### 2. 更新・廃棄の申請タイミング
- 処理区分を「更新」「廃棄」に設定した後、いつ・どのように申請が作成されるか
- 自動で申請が作られるのか、手動でアクションが必要か
- → **要決定**

### 3. 増設・新規申請のリモデル管理画面での扱い
- リモデル管理画面は既存資産の処理区分決定が主目的
- 増設や新規申請（原本にない新しい機器の要望）はどこから行うか
- → **要決定**
