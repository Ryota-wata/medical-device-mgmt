### Step 5: 差分判定（強制出力義務 — 違反 10 を受けて 2026-05-13 強化）

**コミット message に以下の構造化リストを必ず明記すること。リストが無い commit は Step 5 未履行 = 完了報告禁止。**

#### A. Figma の主要セクション一覧（スクショから視覚的に列挙）
スクショを上から順にトポロジカルに分解:
```
1. ヘッダー: [タイトル / アクションボタン]
2. プログレスバー / タブ: [ステップ数 / 現在位置]
3. サブタイトル行: [情報項目]
4. セクション 1: [ヘッダー形式 (色帯/プレーン) / フォーム構造 / フッター]
   4.1 サブセクション...
5. セクション 2: ...
6. サイドバー / プレビューエリア: [タブ式 / 単一エリア / 内容]
7. フッターアクション: [ボタン構成]
```

#### B. 現実装との対応マッピング（各セクション）
```
セクション 1 (Figma) → 現実装 lines XX-YY → 差分: none | minor | MAJOR
  差分内容: [具体的に]
  対応: [適用 | 保留 (理由) | 確認待ち]
セクション 2 ...
```

#### C. 差分の自己判定
- `none`: 文言・配置・色・アイコン・構造すべて一致
- `minor`: 色トークン / padding / border / アイコン形態のみ違い → Step 6 で適用
- `MAJOR`: ヘッダー帯の有無 / セクション分割の違い / サイドバー構造の違い / 列構成の違い
  → **MAJOR が 1 件でもあれば自分で「ほぼ同じ」と判断する権限はない**。必ず:
  - (a) Figma 通りに実装する、または
  - (b) ユーザーに「この構造差分は許容するか」確認する
  - **どちらでもなく「色のみ修正で完了」とは絶対にしない**

#### D. 機能要素チェック (従来項目)
- [ ] 機能要素の追加/削除を行っていないか？
- [ ] テキスト（ラベル・見出し・ボタン文言・プレースホルダー）を現モックから変更していないか？
- [ ] state 変数・ハンドラー・ロジック・遷移先を変更していないか？
- [ ] sticky/fixed/モーダル/条件表示等の振る舞いを維持しているか？

#### E. 色判定の強制プロセス（2026-05-13 違反 11 を受けて追加）
**「Figma パレットトークン使用 = 色 OK」とは絶対に判断しない。具体的な hex 値レベルで Figma と一致させる:**

1. **Figma 主要要素の hex 値を取得**して列挙（コミット message に出力）:
   - get_design_context / get_variable_defs を呼び、CSS variable に紐づく色を取得
   - 列挙対象: ヘッダー帯 bg / セクションヘッダー bg / 内側テーブル `<th>` bg / 各種 pill bg / button bg / ボーダー色

2. **現実装の対応要素の hex を列挙**:
   - 各要素の `bg-*` / `text-*` / `border-*` クラスから現在の色 token を解決

3. **差分テーブル**:
   ```
   | 要素 | Figma hex | 現実装 hex | 一致 |
   |---|---|---|---|
   | セクションヘッダー bg | #XXXXXX | #YYYYYY | ❌ |
   ```

4. **一致しない要素は MAJOR 扱いで適用**:
   - パレットトークン内であっても具体的トークン選択が間違っていれば MAJOR
   - 例: Figma が `surface-screen (#FAFAFA)` を使っているのに `stroke-card (#F1F1F1)` を使っている → MAJOR

#### F. ボタン判定の強制プロセス（同上）
ボタン要素ごとに以下を Figma と並べて確認:

| 観点 | チェック内容 |
|---|---|
| 配置 | 左寄せ/右寄せ/中央/space-between、フッター bar の場合は同じ並びか |
| サイズ | width / height / padding |
| 色 | bg / text / border / hover 状態 |
| 形状 | rounded / outline / fill |
| アイコン | 有無 / 位置 (前/後) / 種類 |
| 文言 | テキストとアイコン併記時の順序 |

**いずれか 1 つでも不一致なら MAJOR**。「機能は同じだから OK」と勝手判断しない。

#### H. 配置・階層判定の強制プロセス（2026-05-13 違反 13 を受けて追加）
**Figma の section hierarchy (どのセクションの中・外) を必ず確認する:**

各要素について以下を Figma スクショから読み取り、現実装と一致させる:
1. **親セクションは何か** (Header の中 / Section の中 / グローバル area)
2. **兄弟要素との順序** (上 / 下 / 隣)
3. **インデント階層** (Section 内のサブセクション / 同階層)

「現実装の位置を温存して中身だけ書き換える」は **位置の MAJOR 差分を見逃す原因**:
- 例: Figma で STEP 1 の中にあるテーブルを、現実装の「STEP 1 の外」のままテーブル化する
- これは現実装の構造を疑わず、書き換え作業に視野を限定した失敗
- Step 6 実装前に必ず「Figma での位置」と「現実装での位置」を比較する

#### I. テーブル構造判定の強制プロセス（2026-05-13 違反 14 を受けて追加）
**テーブル UI を実装する前に Figma の正確な行数 × 列数を読み取る:**

1. **行数**: Figma スクショで縦に何セット並んでいるか数える
2. **列数**: Figma スクショで横に何カラム並んでいるか数える
3. **N 行 × M 列** の正確な値をコミット message に明記する
4. **「同じデータだから 2 行 3 ペアにまとめてもいい」と勝手判断しない**
   - Figma が 5 行 × 2 列なら現実装も 5 行 × 2 列
   - レイアウトの圧縮・展開は MAJOR 差分扱い

「業界の定石」「省スペースのため」「現実装パターンに合わせて」等の理由で
列数・行数を変える権限は実装者にない。Figma が正解。

例:
- Figma: 申請No / 品名 / メーカー / 型式 / 症状 を 5 行 × 2 列 (label | value)
- 誤実装: 2 行 × 6 列 (label | value | label | value | label | value × 3 ペア)
- これは MAJOR 構造差分

#### G. アイコン判定の強制プロセス（同上）
- Figma のアイコンを `get_design_context` で取得
- 該当する lucide-react アイコンが §永久承認済みパターン② のマッピング表に存在するか確認
- マッピング表にない場合 → ユーザー確認必須
- 「似てるから」「同じ意味だから」で勝手に lucide アイコンを選ばない
- アイコンサイズ (16 / 18 / 20 / 24 px) も Figma と合わせる

## 強制 enforcement（2026-05-13 違反 9 を受けて追加）

### 機械的 pre-commit 監査（必須）
コミット前に **必ず** 以下を実行し、日本語テキスト diff を確認:
```bash
git diff --staged <file> | grep -E '^[-+]' | grep -v '^[-+]{3}' | \
  grep -oE '"[^"]*[ぁ-んァ-ヶー一-龯][^"]*"|>[^<]*[ぁ-んァ-ヶー一-龯][^<]*<'
```
- **日本語テキストの追加/削除/変更行が 1 行でも検出 → commit 中止**
- ユーザーに「この文言変更は OK か」事前確認、承認得てから再 commit
- 例外なし。「Figma に合わせるだけ」「Figma が簡素化」「機能温存と矛盾」も例外と認めない

### 「保持できなかった項目」発生時の強制 STOP
Step 9 で「保持できなかった項目」が **1 件でも発生 → 完了報告禁止**:
- 違反 7 で「明示すれば OK」としていたが、これが抜け穴になった
- 「Figma 不在」「機能温存」等の理由付けで省略する権限は私（実装者）には**ない**
- 必ずユーザー判断を仰ぐ（保留として次画面に進む）

### 操作対象の明示
「Figma 構造再現」が許可するのは以下のみ:
- `class` 属性 / `style` / 配置 (flex/grid) / `border` / `padding` / `color` トークン / アイコンライブラリ置換

以下は **触らない** (触りたい場合はユーザー確認が必須):
- 子テキスト（ラベル・見出し・ボタン文言・プレースホルダー・キャプション・注意文）
- 列の追加/削除（colSpan/rowSpan 構造の変更含む）
- セクション見出し（テキスト帯）の削除
- サマリー表示（合計/件数）の削除/追加
- トグル/ボタンの機能変更（クリック挙動）

## 永久承認済みパターン（2026-05-13 追加）

ユーザーから事前承認済で、Figma 反映時に確認不要で適用可。

### ① テーブル UI の縦罫線
**全ての `<table>` UI（基本情報フォーム / データ一覧 / 集計表 etc.）はセル間に縦罫線を表示する。**
- 横罫線のみで縦区切りが無い実装は **NG**（セルの境界が不明瞭になる）
- 実装方法:
  - `<td>` / `<th>` に `border border-stroke-input` （または `border-r`/`border-l` の組み合わせ）を必ず付与
  - `<table>` 全体に `border-collapse` も併用
  - 例外として「フォーム入力欄を兼ねるテーブル（ラベル+入力）」も最低限の縦罫線で列区切りを示すこと
- Figma のテーブルは縦罫線あり前提で設計されている。罫線を省略しない

### ② テキスト → アイコン置換のマッピング承認
以下のマッピングは **テキスト削除してアイコンのみ化を承認**（aria-label に元テキストを保持すること）:

| 元テキスト | 置換アイコン (lucide-react) |
|---|---|
| 編集 | `Pencil` |
| 削除 | `Trash2` |
| 閉じる / × | `X` |
| 戻る | `ArrowLeft` / `ChevronLeft` |
| 次へ / 進む | `ArrowRight` / `ChevronRight` |
| 検索 | `Search` |
| 追加 | `Plus` |
| 並び替え | `ArrowUpDown` |
| ダウンロード / 出力 | `Download` |
| アップロード / 取込 | `Upload` |
| 表示 | `Eye` |
| 非表示 | `EyeOff` |
| 印刷 | `Printer` |
| 確定 / 完了 | `Check` |

**マッピングが上の表に無いテキストは、アイコン化前に必ずユーザー確認**（誤マッピングで意味が変わる事故を防ぐ）。

`aria-label` には元の日本語テキストをそのまま保持すること。例:
```tsx
<button aria-label="編集"><Pencil size={16} /></button>
```
